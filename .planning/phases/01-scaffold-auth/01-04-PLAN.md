# Plan 01-04: Onboarding 流程

**Phase:** 1 — 项目脚手架与认证
**Requirements:** FOUND-01
**Status:** Ready

## Goal

实现首次启动的 Onboarding 认证流程：用户输入 GitHub PAT → 验证 → 存储 → 进入主界面。

## Scope

### apps/desktop-ui/src/routes/onboarding/+page.svelte
1. Onboarding 页面布局
   - 居中卡片式布局（非 Sidebar 模式）
   - 步骤指示器：Step 1 认证 → Step 2 完成
   - geek taste logo + 标题

2. Step 1: Token 输入
   - 密码输入框（带显示/隐藏切换）
   - 帮助链接："如何创建 GitHub PAT?" → 打开 GitHub 设置页（tauri-plugin-shell）
   - "验证并继续" 按钮
   - 加载状态：验证中显示 spinner

3. 验证逻辑
   - 调用 `invoke('validate_github_token', { token })`
   - 成功：显示用户头像 + 用户名确认 → "确认并继续"
   - 失败：显示具体错误消息
     - 401: "Token 无效，请检查是否正确复制"
     - 403: "Token 权限不足，需要 repo 和 user 范围"
     - 网络: "无法连接 GitHub，请检查网络"

4. Step 2: 确认并存储
   - 调用 `invoke('store_github_token', { token })`
   - 调用 auth store 的 `initAuth()`
   - 重定向到 `/`（Home）

5. 路由守卫集成
   - 如果已认证，重定向到 `/`（防止重复 onboarding）

### apps/desktop-ui/src/lib/components/OnboardingCard.svelte
- 可复用的认证卡片组件（也可直接在页面中实现）

### Key Design Decisions
- Onboarding 全屏模式（不显示 Sidebar）
- 使用 `tauri-plugin-shell` 打开外部链接（GitHub PAT 创建页面）
- 验证失败不清空输入（用户可能只是复制不完整）
- 认证成功后 token 立即存储到 keyring
